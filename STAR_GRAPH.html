STAR-GRAPH PLAN · 2026-05-09

star-graph — keyword network across crawled corpus

Trích xuất keyword từ tất cả articles, build co-occurrence network, render interactive graph để khám phá topic trends, source overlap, và discovery path. Phase P8 sau khi Web UI v2 chạy.

Depends on — P5 (DB) + P7 (Web UI) Effort — 4–5 ngày Stack — KeyBERT · SQLite · Cytoscape.js

Concept

Star-graph = đồ thị keyword cộng-xuất-hiện. Node là keyword, edge là quan hệ co-occurrence trong cùng bài; trọng số = mức độ liên kết.

Node

Keyword (term hoặc phrase). Size theo frequency, color theo cluster (Louvain community detection).

Edge

Co-occurrence trong cùng article. Weight theo PMI (Pointwise Mutual Information) chuẩn hóa.

Cluster

Cộng đồng keyword auto-detect — vd. cụm "Kafka / event-driven / Kinesis" tách khỏi cụm "React / SSR / hydration".

Drilldown

Click node → side panel hiện top articles chứa keyword đó, lọc theo source.

Filter

Theo source, time range, min-frequency, min-edge-weight. Real-time prune trên client.

Search

Type-ahead tìm node theo tên, focus + neighbors highlight.

Use cases

  • Topic discovery — thấy cụm đề tài đang hot trong corpus, không cần đọc từng bài
  • Source overlap — Uber và Stripe cùng nói về "PostgreSQL replication" như thế nào, có lệch quan điểm không
  • Bridge concepts — keyword nào nối 2 cluster (vd. "Kafka" nối "event-driven" với "data lake")
  • Reading path — chọn keyword gốc → graph gợi ý sequence bài để đọc dần
  • Dataset slicing — export articles thuộc 1 cluster ra parquet riêng cho fine-tune domain-specific

Pipeline overview

Batch job sau crawl, không inline trong fetcher (giữ crawler nhẹ). Idempotent — chạy lại an toàn.

articles.db (đã có sau P1–P5) │ ▼ ┌─ extract-keywords ──────────────────────────────┐ │ · KeyBERT semantic (top 15/article) │ │ · + tech glossary boost (Kafka, k8s, …) │ │ · normalize: lower, lemma, dedupe variants │ │ · filter: min_score, stopword, blacklist │ └────────────────────────┬────────────────────────┘ │ ▼ keywords · article_keywords (DB) │ ▼ ┌─ build-graph ────────────────────────────────────┐ │ · co-occurrence count cho mọi (kw_i, kw_j) │ │ · PMI normalize (loại bỏ "the the" effect) │ │ · prune: min_co_count, min_pmi │ │ · Louvain community detect → cluster_id │ └────────────────────────┬────────────────────────┘ │ ▼ keyword_edges (DB) + graph_meta │ ▼ GET /graph.json (FastAPI, cached) │ ▼ Cytoscape.js render (browser)

Keyword extraction

Lựa chọn extractor là quyết định quan trọng nhất — quality của graph = quality của keyword. So sánh 4 phương án thực dụng.

OptionQualitySpeedCostMultilingualPick
KeyBERT + sentence-transformers all-MiniLM-L6-v2 Cao — semantic ~50 articles/s CPU Free, model 23MB Khá (en mạnh nhất) đề xuất
YAKE (rule-based) TB — bỏ context ~500/s CPU Free, 0 model Mạnh (en/vi/jp/…) Fallback nhanh
RAKE Thấp ~1k/s Free OK —
LLM (Claude Haiku / gpt-4o-mini) Rất cao ~10/s API limit ~$5–10 / 10k articles Excellent Nếu budget cho phép

Recommended hybrid (Option A)

# pseudo-pipeline per article
kw_keybert = KeyBERT(model).extract(text, top_n=15, ngram=(1,3),
                                    diversity=0.6, stopwords='en')
kw_glossary = match_tech_glossary(text)            # exact match: Kafka, k8s, gRPC, …
kw_merged   = dedupe(kw_keybert + kw_glossary)
kw_final    = filter(kw_merged,
                     min_score=0.35,
                     blacklist=GENERIC_NOISE)      # vd. "team", "company", "engineer"
                     min_chars=2,
                     max_words=4)

Tech glossary (~500 từ) seed từ awesome-* lists + manual curation. Lưu trong configs/glossary/tech.txt, có thể override qua YAML mỗi source.

Normalization pipeline

  • Lowercase nhưng giữ acronym: Kubernetes → kubernetes, K8S → k8s (không lowercase nếu len ≤ 4 và all-caps)
  • Lemmatize: databases → database (spaCy en_core_web_sm, optional)
  • Alias map: k8s ≡ kubernetes, postgres ≡ postgresql — config trong configs/aliases.yaml
  • Stopword: extended list + domain-specific (engineer, team, system…)

DB schema (mở rộng SQLite)

3 bảng mới + 1 view. WAL mode đã có từ P5; FTS5 không cần cho graph.

-- 1. keywords: master list, dedupe alias
CREATE TABLE keywords (
  id          INTEGER PRIMARY KEY,
  term        TEXT NOT NULL UNIQUE,        -- normalized (lower/lemma)
  display     TEXT NOT NULL,                -- pretty form (Kubernetes)
  doc_freq    INTEGER NOT NULL DEFAULT 0,   -- số article chứa keyword này
  source      TEXT NOT NULL,                -- 'keybert' | 'glossary' | 'manual'
  cluster_id  INTEGER,                      -- gán bởi Louvain
  created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX idx_keywords_doc_freq ON keywords(doc_freq DESC);
CREATE INDEX idx_keywords_cluster  ON keywords(cluster_id);

-- 2. article_keywords: many-to-many với score
CREATE TABLE article_keywords (
  article_id  INTEGER NOT NULL REFERENCES articles(id) ON DELETE CASCADE,
  keyword_id  INTEGER NOT NULL REFERENCES keywords(id) ON DELETE CASCADE,
  score       REAL    NOT NULL,             -- KeyBERT cosine similarity 0–1
  is_glossary INTEGER NOT NULL DEFAULT 0,   -- 1 nếu match từ glossary
  PRIMARY KEY (article_id, keyword_id)
);
CREATE INDEX idx_ak_keyword ON article_keywords(keyword_id);
CREATE INDEX idx_ak_article ON article_keywords(article_id);

-- 3. keyword_edges: co-occurrence weighted
CREATE TABLE keyword_edges (
  src_id    INTEGER NOT NULL REFERENCES keywords(id) ON DELETE CASCADE,
  dst_id    INTEGER NOT NULL REFERENCES keywords(id) ON DELETE CASCADE,
  co_count  INTEGER NOT NULL,               -- bài chứa cả 2
  pmi       REAL    NOT NULL,               -- PMI normalized
  PRIMARY KEY (src_id, dst_id),
  CHECK (src_id < dst_id)                  -- undirected: chỉ lưu 1 chiều
);
CREATE INDEX idx_edges_pmi ON keyword_edges(pmi DESC);

-- 4. graph_meta: build run audit
CREATE TABLE graph_meta (
  built_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  n_articles    INTEGER,
  n_keywords    INTEGER,
  n_edges       INTEGER,
  n_clusters    INTEGER,
  config_hash   TEXT
);

-- view: keyword + cluster cho client convenience
CREATE VIEW keyword_full AS
  SELECT k.id, k.term, k.display, k.doc_freq, k.cluster_id,
         (SELECT COUNT(*) FROM keyword_edges WHERE src_id = k.id OR dst_id = k.id) AS degree
  FROM keywords k;

Edge weighting algorithm

Co-count thuần thiên về stopword frequency. PMI fix bias đó nhưng nhạy với keyword hiếm. Chọn PPMI (positive PMI) làm trade-off.

# cho cặp (k_i, k_j):
#   p(i,j) = co_count(i,j) / total_articles
#   p(i)   = doc_freq(i) / total_articles
#   p(j)   = doc_freq(j) / total_articles
#
#   PMI(i,j) = log( p(i,j) / (p(i) * p(j)) )
#   PPMI     = max(0, PMI)        # bỏ negative associations
#
# normalize về [0,1]:
#   NPMI = PMI / (-log(p(i,j)))
#
# pick: NPMI ∈ [0, 1] để weight = NPMI khi > 0
edge_weight = max(0, NPMI(i, j))

Pruning thresholds (config-able):

  • min_doc_freq = 3 — keyword phải xuất hiện ít nhất 3 bài (loại noise)
  • min_co_count = 2 — cặp keyword phải co-occur ít nhất 2 bài
  • min_npmi = 0.15 — chỉ giữ edge có association meaningful
  • max_edges_per_node = 30 — top-30 edges/node để giữ graph readable

Community detection

Sau khi prune, chạy networkx.algorithms.community.louvain_communities trên graph weighted. Gán cluster_id vào keywords. Mỗi cluster sẽ có 1 màu trong viz. Resolution = 1.0 mặc định, có thể tune.

Tip: Sau Louvain, đặt tên cluster bán-tự-động: top-3 keyword theo doc_freq trong cluster làm label. Vd. cluster "Kafka · streaming · backpressure". Lưu vào bảng riêng clusters(id, label, n_keywords).

Visualization — Cytoscape.js

Single CDN script. Force-directed (cose-bilkent layout). Performance OK đến ~5k nodes / 20k edges trên laptop.

Tại sao Cytoscape.js?

  • Declarative styling: node size theo doc_freq, edge width theo npmi, color theo cluster_id — tất cả qua selector
  • Built-in layouts: cose, cose-bilkent, fcose (recommend fcose cho graph dense)
  • Click/hover events natural, dễ wire vào side panel HTMX
  • ~80KB gzipped, không build step

CLI & flow

3 commands mới, idempotent, có flag --source để chạy từng nguồn. Crawl không bao giờ trigger extraction tự động — user chủ động.

# 1. trích keyword cho articles mới (delta)
star-crawl extract-keywords [--source NAME] [--all] [--rebuild]
  --source uber_engineering        # chỉ articles từ source này
  --rebuild                        # xóa article_keywords cũ và làm lại

# 2. build graph từ article_keywords hiện có
star-crawl build-graph [--min-freq 3] [--min-npmi 0.15] [--max-edges-per-node 30]

# 3. inspect (CLI thuần, không Web UI)
star-crawl graph stats              # tổng node/edge/cluster
star-crawl graph top --by degree    # top hub keywords
star-crawl graph cluster 7          # list keywords trong cluster 7
star-crawl graph neighbors kafka    # top neighbors của 1 keyword

# 4. export
star-crawl graph export graphml     # cho Gephi
star-crawl graph export json        # cho frontend dev

Typical workflow sau khi crawl xong:

$ star-crawl run --all                    # crawl
$ star-crawl extract-keywords --all       # ~5–10 phút cho 10k articles
$ star-crawl build-graph                  # ~30s
$ star-crawl serve                        # Web UI có /graph live

Risks & mitigations

RiskLikelihoodMitigation
KeyBERT chậm trên CPU lớn (10k articles × 15 keyword) Cao Batch encode, multiprocessing, cache embedding theo content_hash để rerun không tính lại
Tech glossary lệch theo time (term mới: "agentic", "MCP" 2026) TB Glossary file rời, dễ update; KeyBERT semantic catch term mới qua embedding
Graph quá dense → render lag (>10k edges sau prune) TB Tăng min_npmi, hard cap max_edges_per_node, dùng haystack edge style
Multilingual articles (Grab có chunk vi, jp, …) Thấp (đa số en) Detect lang trước extract; YAKE fallback cho non-en
Keyword "noise" (engineer, system, team, post) Cao Blacklist + min_doc_freq + manual review top-100 sau lần đầu build
Cluster label kém (top-3 freq không đại diện) TB Manual override qua configs/cluster_labels.yaml sau khi inspect lần đầu
Tăng size articles.db đáng kể Thấp article_keywords ~10–15 dòng/article = ~150k rows = vài MB; OK

Open questions

  1. Extractor: bắt đầu với KeyBERT (đề xuất), YAKE (cho speed), hay LLM (cho quality cao nhất, ~$5–10)?
  2. Glossary scope: chỉ tech terms, hay mở rộng ra business/process (CI/CD, postmortem, SLA, OKR)?
  3. Aliases: tự build từ corpus (BERT cluster gần) hay manual file? Mỗi cách có pros riêng.
  4. Time dimension: muốn graph thay đổi theo thời gian (animation slider) không? Effort +1d nhưng useful cho trend analysis.
  5. Graph viz library: chốt Cytoscape.js, hay muốn thử Sigma.js (faster cho >10k nodes) / D3 (full control, code nhiều hơn)?
  6. Privacy: graph có expose ra public không? Nếu có, scrub author name, redact nội bộ-only keyword.
